AsyncWebRL: Efficient Multi-Step RL for Visual Web Agents

Spencer Whitehead2, Aviral Kumar3, Tong Zhang1

Abstract

Training vision-language web agents with multi-step RL is compute-intensive, with two dominant forms of inefficiency: idle GPUs in synchronous RL, and trajectories that use more steps and tokens than necessary. We present AsyncWebRL, which addresses both.

On the system side, an asynchronous design overlaps rollout, gradient update, and policy refresh across iterations, paired with two web-agent-specific adaptations — an everlasting rollout pool and lightweight screenshot handling — that together deliver up to a 2.9× end-to-end training-throughput speedup over the previously fastest open synchronous pipeline (WebGym).

On the algorithmic side, we identify the per-trajectory normalizer $1/|\tau_i|$ in multi-step GRPO as the root cause of trajectory-level and token-level inefficiency: because failures are systematically longer than successes, it down-weights the negative gradient on failed tokens, so the policy keeps producing verbose memory schemas. Replacing $1/|\tau_i|$ with a constant $1/k$ breaks this coupling, contracting trajectories while preserving aggregate success.

Together, these contributions set a new open-source state of the art on the WebGym out-of-distribution test split (+5.8% relative over the 42.9% prior best), with the largest gains on the harder slices (+42% relative on Medium, +48% relative on Hard).

System: Asynchronous Multi-Step RL

Two web-agent-specific changes turn WebGym's synchronous rollout pool into an asynchronous pipeline that keeps both rollout workers and the trainer busy:

Everlasting rollout pool. Rollout workers stay continuously alive across iteration boundaries. When one episode ends, the next begins immediately on the same worker without waiting for the rest of the batch or the next iteration. The policy parameter update of $\pi_\theta$ can happen at any time while rollout continues; new weights are broadcast in place to the inference workers and the next rollout segment is sampled under the updated policy.

Lightweight screenshot handling. Per-step image tensors no longer transit the shared inter-worker object store. A dedicated Ray actor (PixelStore) holds screenshots keyed by an opaque ID; only the key passes through RPC. The trainer fetches the tensor lazily when it needs to construct a forward-pass batch.

Decoupled off-policy correction. Asynchronous execution shifts the sampling distribution: the policy that generated a given token, $\pi_{\mathrm{behave}}$, is several updates behind $\pi_\theta$. We adopt the decoupled-PPO factorization: split the importance ratio into a rollout-staleness factor $\pi_{\mathrm{prox}}/\pi_{\mathrm{behave}}$ (unclipped weight) and a current-update factor $\pi_\theta/\pi_{\mathrm{prox}}$ (the only factor that gets clipped). Empirically this roughly halves the clip rate.

Algorithm: Replacing $1/|\tau_i|$ with $1/k$

Multi-step GRPO assigns a per-trajectory normalizer $1/|\tau_i|$ to the step-level advantage, so trajectories with more steps contribute smaller per-token gradients. Because failures are systematically longer than successes in WebGym, this normalizer silently down-weights the negative gradient on the failure modes the policy needs to unlearn. The result: the policy keeps producing verbose memory schemas and the failure rate plateaus.

Replacing $1/|\tau_i|$ with a constant $1/k$ (we use $k=10$) decouples the per-token gradient magnitude from trajectory length. Long failed trajectories now contribute equally per step to the gradient as short successful ones, so the policy is forced to shorten them. Empirically this contracts trajectories while preserving aggregate success, and turns the previously declining reward curve into a monotone climb.
